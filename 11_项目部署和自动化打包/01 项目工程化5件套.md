# 一、Husky9 + Commitlint + Commitizen + lint-staged

- **Husky9**：Git 钩子工具，在**提交前、写备注前**自动执行命令
- **Commitlint**：强制校验 Git 备注是否规范，不规范不让提交
- **Commitizen**：交互式引导你写规范备注，不用记格式
- **lint-staged**：只检查本次修改的代码，自动格式化
- **Prettier**：代码格式化工具

# 二、环境准备

- 已初始化 Git 项目：`git init`
- 已初始化 package.json：`npm init -y`
- Node.js >= 18

# 三、正式开始搭建

## 1. 安装所有依赖

```bash
npm install -D husky @commitlint/cli @commitlint/config-conventional commitizen cz-conventional-changelog lint-staged prettier
```

### ✅ 作用说明

- `husky`：最新版 Git 钩子管理
- `@commitlint/xxx`：commit 备注校验
- `commitizen / cz-xxx`：交互式规范提交
- `lint-staged`：只校验修改的文件
- `prettier`：代码格式化

## 2. 初始化 Husky

```bash
npx husky init
```

### ✅ 作用说明

- 自动创建 `.husky/` 目录
- 自动在 `package.json` 添加 `prepare` 脚本
- 让 Git 克隆项目后**自动启用钩子**

执行后你会看到：

```json
// package.json
// package.json 自动生成
"scripts": {
  "prepare": "husky"
}
```

## 3. 创建 Commitlint 配置文件

**根目录新建：`commitlint.config.js`**

```js
// 让 commitlint 遵循 Angular 官方提交规范
module.exports = {
  extends: ['@commitlint/config-conventional']
}
```

### ✅ 作用说明

定义 `feat/fix/docs` 等规范，不满足就拦截提交。


## 4. 配置 Commitizen 交互式提交

打开 `package.json`，在**最底部**添加：

```json
"config": {
  "commitizen": {
    "path": "cz-conventional-changelog"
  }
}
```

### ✅ 作用说明

让 `npx cz` 命令可以弹出交互式选择框，引导写规范备注。


## 5. 配置 lint-staged（只格式化修改的文件）

**根目录新建：`.lintstagedrc.js`**

```js
module.exports = {
  "*.{js,ts,jsx,tsx,vue,css,scss,md}": [
    "prettier --write" // 自动格式化
  ]
}
```

### ✅ 作用说明

只对本次 `git add` 的文件执行格式化，速度极快，不污染无关代码。

# 四、配置 Git 钩子（核心！最新 Husky 写法）

## 钩子 1：提交前自动格式化代码

**创建文件：`.husky/pre-commit`**

内容直接写：

```bash
npx lint-staged
```

### ✅ 作用说明

每次执行 `git commit` 前自动跑：代码格式化。

## 钩子 2：校验 commit 备注是否规范

**创建文件：`.husky/commit-msg`**

内容直接写：

```bash
npx --no -- commitlint --edit $1
```

### ✅ 作用说明

每次写备注后，自动校验是否规范，不规范直接**拒绝提交**。

# 六、规范提交格式（中文对照表）

```palintext
feat: 新增功能
fix: 修复 bug
docs: 修改文档
style: 格式（不影响代码逻辑）
refactor: 重构（既不是功能也不是bug）
test: 测试代码
chore: 构建/工具配置
```

示例：

```palintext
feat: 新增用户登录页面
fix: 修复首页崩溃问题
docs: 更新部署教程
```

# 九、EditorConfig


## 9.1 在项目根目录新建 `.editorconfig` 文件

直接复制下面全部内容（前端通用顶配）：

```ini
# 标记这是项目根配置，编辑器不会再向上找父级配置
root = true

# 所有文件全局通用规则
[*]
charset = utf-8                  # 统一编码 UTF-8，避免乱码
indent_style = space             # 缩进用空格，不用 tab（前端通用标准）
indent_size = 2                  # 每次缩进2个空格（JS/TS/Vue主流）
end_of_line = lf                 # 统一换行符 LF（跨Windows/Mac/Linux防冲突）
trim_trailing_whitespace = true  # 自动删除行尾多余空格
insert_final_newline = true      # 每个文件末尾强制空一行，符合通用规范

# 专属适配：后端/静态资源特殊规则
[*.{js,ts,jsx,tsx,vue,json,html,css,scss,less}]
indent_size = 2

# Python 后端常用4空格缩进
[*.py]
indent_size = 4

# Makefile 必须用 tab，不能用空格
[Makefile]
indent_style = tab
```

## 9.2 每一行「为什么这么写」详细解释

1. `root = true`

> 告诉编辑器：这是项目最顶层配置，不再去上级文件夹读其他配置，防止多层配置打架。

2. `charset = utf-8`

> 全局统一编码，杜绝 Windows 打开中文乱码、接口报文编码不一致问题。

3. `indent_style = space / indent_size = 2`

> 前端行业标准：所有代码缩进 2 个空格；杜绝有人用 Tab、有人用 4 空格，导致 Git 全是空格冲突。

4. `end_of_line = lf`

> Windows 默认 CRLF 换行，Mac/Linux 是 LF；统一 LF，跨系统协作 Git 永远不会报「换行符冲突」。

5. `trim_trailing_whitespace = true`

> 自动删掉行尾多余空格，肉眼看不见但 Git 会标红冲突的脏空格。

6. `insert_final_newline = true`

> 很多编译工具、Git、CI 流水线要求文件末尾空一行，不加会报警告甚至编译失败。


## 9.3 编辑器必须开启生效（关键）

1. VS Code：安装插件 **EditorConfig for VS Code**（官方插件）
2. 安装后重启编辑器，打开项目，配置自动覆盖你本地个人设置
3. 哪怕换电脑、换编辑器，打开代码格式永远一致

## 9.4 五件套完整层级 & 分工（彻底搞懂）

| 工具                        | 管控层级     | 核心作用                                     |
| ------------------------- | -------- | ---------------------------------------- |
| **EditorConfig**          | 编辑器底层    | 统一缩进、换行、编码、行尾空格（打开代码就规范）                 |
| **Prettier**              | 代码格式化    | 格式化引号、分号、换行排版（精细美化代码）                    |
| **lint-staged**           | Git 暂存区  | 只格式化你本次改的文件，提速不污染全局                      |
| **Husky**                 | Git 钩子门禁 | 在提交 / 推送节点自动触发所有校验                       |
| **Commitlint+Commitizen** | 提交备注     | 引导 + 强制规范 Git 提交日志，方便复盘、打版本、生成 CHANGELOG |

## 9.5 全流程最终闭环（你日常写代码的完整链路）

1 写代码 → EditorConfig 强制编辑器基础格式

2 保存代码 → Prettier 自动排版

3 git add 暂存 → lint-staged 只格式化改动文件

4 提交用 `npx cz` → 交互式写规范备注

5 Husky 触发钩子 → Commitlint 校验备注，不合格直接拦截

6 全部合格 → 允许提交到本地 / 推送到远程
