> 通过一个形象的表达：审查员的工作手册

# 一、结构性类型系统（鸭式辨型）

> “如果它走起路来像鸭子，叫起来像鸭子，那么它就是鸭子。”  
> 审查员不关心你这个对象的“名字”（类型名称）是什么，他只关心“结构”（有哪些属性，属性是什么类型）。

```ts
interface Duck {
  quack(): void;
}

interface Person {
  quack(): void;
}

function makeSound(duck: Duck) {}

const myPerson: Person = { quack: () => console.log("Quack!") };
makeSound(myPerson); // ✅ 审查通过！
```

审查员检查：`myPerson` 有没有 `quack` 方法，并且类型是否匹配？有！那好，尽管你是个 `Person`，但你的结构符合 `Duck` 的合同，我允许你通过

# 二、类型兼容性（核心规则）

这是审查员判断“A 类型是否能赋值给 B 类型”的终极法则。简单来说：

- **对于对象**：如果 A 拥有 B 的所有**必需**属性，并且这些属性的类型是兼容的，那么 A 就兼容 B。
    
    - 也就是说，A 可以比 B 多属性，但不能少。
        
- **对于函数**：情况稍微复杂，但大体是“参数可以少，但不能多；返回值必须匹配”。

# 三、编译时检查，而非运行时

这是最关键的一点！**这个审查员只在你把代码提交去“印刷”（编译）之前工作。一旦代码被编译成 JavaScript 并开始运行，这个审查员就下班了。**

```ts
const something: any = "This is a string";
const length: number = something; // ✅ 编译时通过，因为 any 是“万能通行证”，审查员不管。

// 但运行时，something 可能根本不是字符串，导致 length 得到 undefined，从而在后续代码中出错。
```

所以，`any` 类型就像是给了审查员一个“别多管闲事”的指令，他虽然放行了，但风险自负。

# 四、类型擦除

审查员工作完后，他会把所有类型注解（那些 `: string`, `: number`）都用橡皮擦掉。最终生成的 JavaScript 代码是干干净净、没有任何类型信息的。因为 JavaScript 引擎看不懂这些“合同条款”。

```ts
// TypeScript 输入
let message: string = "hello";
// JavaScript 输出  
let message = "hello";
```

# 五、总结

你可以把 TypeScript 的类型检测想象成：

- **一个极其严格、但只在设计阶段工作的“架构师”或“合同审查员”。**
    
- 他拿着一份由你（通过类型注解或推断）制定的**蓝图**或**合同**。
    
- 他的工作是确保你的**代码实现**和你的**设计蓝图**在**结构上**是匹配的。
    
- 他通过**鸭式辨型**来判断兼容性，而不是死板地看名字。
    
- 他的所有检查都在**代码运行之前**完成，一旦编译通过，他就功成身退，生成的 JavaScript 里不再有他的痕迹。
