# 一、认识预渲染

认识预渲染

 - **默认情况下**，Next.js 会 **预渲染** 每个页面，即预先为每个页面生成 HTML 文件，而不是由客户端 JavaScript 来完成。
 - 预渲染可以带来更好的性能和 SEO 效果。
 - 当浏览器加载一个页面时，页面依赖JS代码将会执行，执行JS代码后会激活页面，使页面具有交互性。(此过程称Hydration)

Next.js 具有两种形式的预渲染：

 - **静态生成(推荐)**：HTML 在 构建时 生成，并在每次页面请求（request）时重用。
 - **服务器端渲染**：在 每次页面请求（request）时 重新生成 HTML页面。

> [!tip] 提示：
> 出于性能考虑，相对服务器端渲染，更 **推荐** 使用 **静态生成** 。

# 二、静态生成（SSG）

静态生成（也称SSG 或 静态站点生成）

 - 如果一个页面使用了 **静态生成**，在 **构建时** 将生成此页面对应的 HTML 文件 。这意味着在生产环境中，运行 next build 时将生成该页面对应的 HTML 文件。然后，此 HTML 文件将在每个页面请求时被重用，还可以被 CDN 缓存。
 - 在 Next.js 中，你可以静态生成 **带有或不带有数据** 的页面。

## 2.2 带数据

当某些页面需要获取外部数据以进行预渲染，通常有两种情况：

1. 情况一：页面 **内容** 取决于外部数据：直接在 `async` 函数组件中获取数据。

```tsx
// app/posts/page.tsx
// 定义文章的数据结构
interface Post {
  id: number;
  title: string;
  body: string;
}

export default async function PostsPage(): Promise<JSX.Element> {
  // 明确 fetch 返回的数据类型
  const res = await fetch('https://api.example.com/posts');
  const posts: Post[] = await res.json();

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

2. 情况二：页面 **paths（路径）** 取决于外部数据`export async function generateStaticParams()`

```tsx
// app/posts/[id]/page.tsx
// 定义文章的数据结构
interface Post {
  id: number;
  title: string;
  body: string;
}

// generateStaticParams 的类型：返回一个参数对象数组
export async function generateStaticParams(): Promise<{ id: string }[]> {
  const res = await fetch('https://api.example.com/posts');
  const posts: Post[] = await res.json();

  return posts.map((post) => ({
    id: post.id.toString(),
  }));
}

// 页面组件的 props 类型
interface PageProps {
  params: Promise<{ id: string }>; // Next.js 16 中 params 是 Promise
}

export default async function PostPage({ params }: PageProps): Promise<JSX.Element> {
  const { id } = await params; // 必须 await 获取参数
  const res = await fetch(`https://api.example.com/posts/${id}`);
  const post: Post = await res.json();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}
```

> [!tip]
> 类型导出：generateStaticParams 的类型可省略，但显式写出有助于代码可读性和类型检查。
> PageProps 中的 params：在 Next.js 16 中，params 是 Promise，因此需要 await 才能获取参数值。若使用 Next.js 15 或更早版本，params 是同步对象，类型应为 { params: { id: string } }。

## 2.3 静态生成应用场景

建议尽可能使用静态生成（无论有 与 没有数据），因为静态生成的页面可以构建一次，并可由 CDN 提供服。

我们可以为多种类型的页面使用静态生成，包括：

 - 营销页面、官网网站
 - 博客文章、投资组合
 - 电子商务产品列表、帮助和文档

> [!note] 如果在用户请求之前就可以预渲染页面，那么应该选择静态生成。反之，静态生成就不合适了。

例如，页面要显示经常更新的数据，并且页面内容会在每次请求时发生变化，这时可以这样选择：

 - 静态生成(SSG)与客户端数据获取(CSR)结合使用：
	 - 我们可以跳过预呈现页面的某些部分，然后**使用客户端 JS 来填充它们**，但是客户端渲染是不利于SEO优化的，例如：
	 - **在useEffect 中获取数据，在客户端动态渲染页面**。
 - 服务器端呈现（也称动态呈现-SSR）：
	 - Next.js 会根据每个请求预呈现一个页面。缺点是稍微慢一点，因为页**面无法被 CDN 缓存**，但预渲染页面将始终是最新的。

# 三、服务器端渲染(SSR)

服务器端渲染（也称SSR 或 动态渲染）

 - 如果页面使用的是 服务器端渲染，则会在 每次页面请求时 重新生成页面的 HTML 。
 - 要对页面使用服务器端渲染，需要使用异步函数组件。
 - 服务器将在每次页面请求时调用此函数。

```test
app/
└── posts/
    └── [id]/
        └── page.tsx   → 动态路由页面
```

```tsx
import { notFound } from 'next/navigation';

// 定义文章的数据类型
interface Post {
  id: number;
  title: string;
  body: string;
}

// 页面组件的 props 类型
interface PageProps {
  params: Promise<{ id: string }>;
  searchParams: Promise<{ [key: string]: string | string[] | undefined }>;
}

// 强制动态渲染：每次请求都重新生成页面（类似 getServerSideProps）
export const dynamic = 'force-dynamic';

export default async function PostPage({ params, searchParams }: PageProps) {
  // 等待参数解析（Next.js 16 中 params/searchParams 均为 Promise）
  const { id } = await params;
  const query = await searchParams; // 可选的查询参数，如 ?version=2

  // 每次请求都会执行这里的 fetch，因为 dynamic = 'force-dynamic'
  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`, {
    // 可选：显式禁用缓存，确保每次请求都获取最新数据
    cache: 'no-store',
  });

  if (!res.ok) {
    // 如果文章不存在，返回 404
    notFound();
  }

  const post: Post = await res.json();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
      {query.version && <p>版本号：{query.version}</p>}
    </article>
  );
}
```

## 🔑 关键点说明

| 项目                                           | 说明                                                                                 |
| -------------------------------------------- | ---------------------------------------------------------------------------------- |
| **`export const dynamic = 'force-dynamic'`** | 强制页面为动态渲染，每次请求都会重新生成 HTML，相当于旧版 `getServerSideProps` 的效果。                          |
| **`fetch` 的 `cache: 'no-store'`**            | 确保每次请求都从源服务器获取最新数据，不读取任何缓存。如果未指定，Next.js 默认会尝试缓存数据（在静态生成时）。                        |
| **`params` 和 `searchParams`**                | 在 Next.js 16 中，这两个参数都是 **Promise**，需要使用 `await` 解构。`searchParams` 可用于处理 URL 查询字符串。 |
| **`notFound()`**                             | 从 `next/navigation` 导入，用于在数据不存在时触发 404 页面。                                         |
| **TypeScript 类型**                            | 定义了 `Post` 接口、`PageProps` 类型，确保类型安全。                                               |
## 📦 可选的其他配置

如果你希望页面在某些情况下静态生成，其他情况下动态渲染，可以调整 `dynamic` 和 `fetch` 缓存策略的组合：

- **`export const dynamic = 'auto'`**（默认） – 尽可能静态生成，但允许在特定条件下动态渲染。
- **`export const revalidate = 0`** – 禁用静态生成，相当于 `force-dynamic` 的替代写法。
- **`fetch` 中使用 `next: { revalidate: 10 }`** – 设置增量静态再生（ISR），允许页面静态生成，但每隔 10 秒重新验证一次。

## 💡 总结

在 App Router 中实现服务器端渲染（SSR）非常直接：

1. 创建一个异步的 Server Component 页面
2. 设置 `export const dynamic = 'force-dynamic'` 或 `export const revalidate = 0`。
3. 在组件内使用 `fetch` 获取数据，并根据需要设置缓存策略。
4. 利用 `params`、`searchParams` 访问路由参数和查询参数。

# 四、增量静态再生（ISR）

**ISR** 是 Next.js 的一项功能，允许你在静态生成页面后，在后台定期重新验证并更新这些页面，而无需重新构建整个应用。它结合了静态站点生成（SSG）和服务器端渲染（SSR）的优点：**增量静态再生（Incremental Static Regeneration），简称（ISR）。**

- 页面首次访问时返回静态 HTML（快速加载）。
- 经过指定时间后，下次访问会触发后台重新生成新版本的页面。
- 更新完成后，新访问者将看到最新内容，旧版本在后台更新期间仍可访问（**stale-while-revalidate**）。

在 App Router 中，ISR 的实现方式非常直观，主要通过 **页面级配置** 或 **数据获取配置** 来完成。


## 方式一：页面级配置

在页面文件中导出 `revalidate` 变量，指定页面的重新验证时间（秒）。该页面所有数据获取（无论使用 `fetch` 还是其他库）都会共享这个重新验证周期。

```tsx
// app/posts/page.tsx
import { notFound } from 'next/navigation';

interface Post {
  id: number;
  title: string;
  body: string;
}

// 设置 ISR：每 60 秒重新验证一次
// revalidata - 重新校验
export const revalidate = 60;

export default async function PostsPage() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts');
  const posts: Post[] = await res.json();

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

## 方式二：数据获取级配置

在 `fetch` 中通过 `next.revalidate` 选项为特定数据请求设置重新验证时间。这种方式可以精细化控制不同数据的缓存策略。

```tsx
// app/posts/page.tsx
interface Post {
  id: number;
  title: string;
  body: string;
}

export default async function PostsPage() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts', {
    next: { revalidate: 60 } // 数据每 60 秒重新验证
  });
  const posts: Post[] = await res.json();

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

> [!tip] **注意**：
> 当页面同时存在页面级 `revalidate` 和 `fetch` 的 `next.revalidate` 时，它们会共同作用，实际行为取决于更细粒度的设置。通常页面级配置会影响整个页面的 ISR 周期。

## 📄 动态路由 + ISR 示例

对于动态页面（如博客详情页），可以使用 `generateStaticParams` 预先生成所有路径，同时通过 `revalidate` 让这些页面定期更新。

```tsx
// app/posts/[id]/page.tsx
import { notFound } from 'next/navigation';

interface Post {
  id: number;
  title: string;
  body: string;
}

// 页面级 ISR：每 120 秒重新验证一次
export const revalidate = 120;

// 构建时预生成所有文章的路径
export async function generateStaticParams() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts');
  const posts: Post[] = await res.json();

  return posts.map((post) => ({
    id: post.id.toString(),
  }));
}

interface PageProps {
  params: Promise<{ id: string }>;
}

export default async function PostPage({ params }: PageProps) {
  const { id } = await params;

  // 这里使用了 fetch，但也可以使用其他 HTTP 客户端
  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`, {
    // 可选：单独设置此数据的缓存时间
    // next: { revalidate: 120 }
  });

  if (!res.ok) {
    notFound();
  }

  const post: Post = await res.json();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}
```

## 🔍 ISR 的工作流程

1. **首次请求**：如果页面不存在于缓存中，Next.js 会生成静态页面并缓存，然后返回给用户。
2. **后续请求**：在缓存有效期内，直接返回缓存的静态页面。
3. **过期后的首次请求**：当超过 `revalidate` 时间后，用户请求会触发后台重新生成。在此期间，用户仍然看到旧的缓存页面（stale）。
4. **重新生成完成**：新生成的页面替换旧缓存，下一次请求将看到更新后的内。


## 📦 使用场景与注意事项

- **适用场景**：内容不频繁更新但需要实时性（如博客文章、产品目录）；大量页面需要静态化但可能修改。
- **性能**：ISR 只在后台按需重新生成页面，避免了全量构建，非常适合大型站点。
- **数据一致性**：如果页面依赖多个数据源，确保它们的 `revalidate` 时间设置合理，避免部分数据过时。
- **与 `fetch` 的区别**：如果你在 Server Component 中使用 `fetch` 且未设置 `cache` 或 `next.revalidate`，Next.js 默认会缓存数据（除非数据在动态函数或动态配置下）。显式设置 `next.revalidate` 是控制 ISR 的最佳实践。


# 五、客户端渲染(CSR)

**客户端渲染** 是指页面的 HTML 生成和数据获取完全在浏览器中完成。服务器只返回一个基本的 HTML 框架（通常是一个空的 `<div id="root">`）和 JavaScript 文件，然后由 JavaScript 在客户端动态填充内容。在 Next.js 中，CSR 通常用于交互性强、对 SEO 要求不高或需要实时更新的页面。

在 Next.js 16 的 App Router 中，要实现 CSR，你需要：

1. 在组件文件顶部添加 `"use client"` 指令，标记该组件为客户端组件。
2. 在组件中使用 React 的 `useEffect`、`useState` 或第三方数据获取库（如 SWR、React Query、axios）来获取数据并管理状态。
3. 根据需要处理加载状态、错误状态等。

```tsx
// app/dashboard/page.tsx
'use client';

import { useState, useEffect } from 'react';
import axios from 'axios';

// 数据类型定义
interface DashboardData {
  visits: number;
  users: number;
  revenue: number;
}

export default function DashboardPage() {
  const [data, setData] = useState<DashboardData | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    const fetchData = async () => {
      try {
        const response = await axios.get<DashboardData>('/api/dashboard');
        setData(response.data);
      } catch (err) {
        setError('Failed to load dashboard data');
        console.error(err);
      } finally {
        setLoading(false);
      }
    };

    fetchData();
  }, []);

  if (loading) return <div>Loading dashboard...</div>;
  if (error) return <div>Error: {error}</div>;
  if (!data) return null;

  return (
    <div>
      <h1>Dashboard</h1>
      <p>Visits: {data.visits}</p>
      <p>Users: {data.users}</p>
      <p>Revenue: ${data.revenue}</p>
    </div>
  );
}
```

## 🚀 使用 SWR 优化数据获取

SWR 是 React Hooks 库，专为数据获取设计，支持缓存、重新验证、焦点重试等功能，非常适合 CSR。

```tsx
// app/profile/page.tsx
'use client';

import useSWR from 'swr';
import axios from 'axios';

interface User {
  id: number;
  name: string;
  email: string;
}

// 定义 fetcher 函数
const fetcher = (url: string) => axios.get(url).then((res) => res.data);

export default function ProfilePage() {
  const { data: user, error, isLoading } = useSWR<User>('/api/user', fetcher, {
    revalidateOnFocus: true, // 窗口重新获得焦点时重新验证
    refreshInterval: 30000,  // 每 30 秒自动刷新
  });

  if (isLoading) return <div>Loading profile...</div>;
  if (error) return <div>Failed to load profile</div>;
  if (!user) return null;

  return (
    <div>
      <h1>{user.name}</h1>
      <p>Email: {user.email}</p>
    </div>
  );
}
```
