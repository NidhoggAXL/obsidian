# 一、JavaScript代码执行

JavaScript代码下载好之后，是如何一步步被执行的呢?

我们知道，浏览器内核是由两部分组成的，以webkit为例:

* WebCore:负责HTML解析、布局、渲染等等相关的工作;
* JavaScriptCore:解析、执行JavaScript代码;

另外一个强大的JavaScript引擎就是V8引擎。


# 二、V8引擎的执行原理

我们来看一下官方对V8引擎的定义:

* V8是用 **C++** 编写的Google开源**高性能JavaScript和WebAssembly引擎**，它用于Chrome和Node.js等
* **快平台性** - 它实现ECMAScript和WebAssembly，并在Windows 7或更高版本，macOs 10.12+和使用x64，IA-32，ARM或MIPS处理器的Linux系统上运行。
* **V8可以独立运行，也可以嵌入到任何C++应用程序中。**

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/173072882300052ywyq.png)

我们到最后的 bytecode（字节码） 的时候还**并不是二进制的**，**只是一种 CPU 可以知道要如何处理的代码编码格式。**

但代码中存在多次调用同种类型的时候就会在 **bytecode**（字节码）的时候经过 **TurboFan** 优化为 **MachineCode**（优化的机器码）来运行得到结果，**这样就不用每次调用的时候都要经过 JavaScript 源代码到 bytecode 字节码的过程**

例如：

```js
function fool(num1, num2) {
	return result = num1 + num2
}

fool(10, 20)
fool(20, 30)
fool(40, 50)
fool(10, 80)
```

> 这样多次调用fool，且每次传入的都是 number 类型的数据的时候就会进行一个 MachineCode 的优化。

**但是**，如果下次传入的实参**不再是 number 类型**的话，就会进行 **Deoptimization**（反优化）操作，回到 bytecode（字节码）的过程，这样就**没有优化了。**

**为了避免这样影响性能的事情发生，在开发中调用函数的时候每次调用尽量保证传入相同类型的数据（这也表现了 JavaScript 的动态类型）**

# 三、V8引擎的架构
V8引擎本身的源码非常复杂，大概有超过100w行C++代码，通过了解它的架构，我们可以知道它是如何对JavaScript执行的

**Parse** 模块会将JavaScript代码转换成AST(抽象语法树)，这是因为解释器并不直接认识JavaScript代码;

* 如果函数没有被调用，那么是不会被转换成AST的;
* Parse的V8官方文档:https://v8.dev/blog/scanner

**lgnition** 是一个解释器，会将AST转换成ByteCode(字节码)

* 同时会收集TurboFan优化所需要的信息(比如函数参数的类型信息，有了类型才能进行真实的运算);
* 如果函数只调用一次，Ignition会解释执行ByteCode;
* lgnition的V8官方文档:https://v8.dev/blog/iqnition-interpreter

**TurboFan** 是一个编译器，可以将字节码编译为CPU可以直接执行的机器码:

* 如果一个函数被多次调用，那么就会被标记为**热点函数**，那么就会**经过TurboFan转换成优化的机器码，提高代码的执行性能,**
* 但是，**机器码实际上也会被还原为ByteCode(字节码)**，这是因为如果后续执行函数的过程中，**类型发生了变化(比如sum函数原来护行的是number类型，后来执行变成了string类型)**，之前优化的机器码并不能正确的处理运算，就会逆向的转换成字节码;
* TurboFan的V8官方文档:https://v8.dev/blog/turbofan-jit
