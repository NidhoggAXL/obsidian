# ``#define``定义宏
> define 机制包话了一个规定，允许把参数替难到文本中，这种实现通常称为宏(macr)或定义去(define macr).

下面是宏的申请方式：
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1709984594000ptx2tv.png)

例如：
```C
#define ADD(X) X+X
int main()
{
	int ret = ADD(5);
	printf("%d\n", ret);
	return 0;
}
```
打印的结果就是10，这里就是利用了宏，把ADD(5)利用宏进行了5+5的运算。

再看下面的代码：
```c
#define SQUARE(X) X*X
int main()
{
	int ret = SQUARE(5+1);
	printf("%d\n", ret);
	return 0;
}
```
这里运行后的结果不是36而是11，那这结果是怎么来的呢？
而是把X替换为了5+1，而不是5+1的和6，这个时候计算的过程是``5+1*5+1``就变成了11了。
**所以要记住：宏是替换，而不是传参。**

如果要让结果是36，那代码就需要改进：
```C
#define SQUARE(X) (X)*(X)
int main()
{
	int ret = SQUARE(5+1);
	printf("%d\n", ret);
	return 0;
}
```

在看下面代码：
```C
#define DOUBLE(X) (X)+(X)
int main()
{
	int a = 5;
	int ret = 10 * DOUBLE(a);
	printf("%d\n",ret);
	return 0;
}
```
这个结果一看就是100，但是结果真的是这样吗？
运行代码会发现结果55，那这又是为什么呢？
当我们带进去替换你会发现``ret = 10 * (5) + (5)``这样的一个结果，显然这不是要的结果，那就改进代码：
```C
#define DOUBLE(X) ((X)+(X))
int main()
{
	int a = 5;
	int ret = 10 * DOUBLE(a);
	printf("%d\n",ret);
	return 0;
}
//这样后打印的结果就是100了
```

> [!tip] 定义宏注意事项：
> 1. 在定义宏的时候，要定义的变量和整体都加括号
> 2. 也不需要加冒号
> 3. 所以用于对数值表达式进行求值的宏定义都应该用这种方式加上括号，避免在使用宏时由于参数中的操作符或邻近操作符之间不可预料的相直作用

# ``#define`` 替换规则
在程序中扩展#define定义符号和宏时，需要涉及几个步骤。
1. 在调用宏时，首先对参数进行检查，看看是否包含任何由#define定义的符号。如果是，它们首先被替换。![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17099860250005a2km4.png)

2. 替换文本随后被插入到程序中原来文本的位置。对于宏，参数名被他们的值替换。
3. 最后，再次对结果文件进行扫描，看看它是否包含任何由#define定义的符号。如果是，就重复上述处理过程。


**注意 :**
1. 宏参数和#define 定义中可以出现其他#define定义的变量。但是对于宏，不能出现递归。
2.  当预处理器搜索#define定义的符号的时候，字符串常量的内容并不被搜索。

# ``#和##``

# 带副作用的宏参数
当宏参数在宏的定义中出现超过一次的时候，如果参数带有副作用，那么你在使用这个宏的时候就可能出现危险，导致不可预测的后果。副作用就是表达式求值的时候出现的永久性效果。例如:
```C
int main()
{
	int a=1;
	int b=a+1;//代码1
	int b=++a;//代码2
	return 0;
}
```
虽然代码1和代码2中b都是2，但是代码2中a也发生了变化，变为2。
这个就是代码2带有副作用，代码1不带副作用。

看一下下面这个代码：
```C
#define MAX(X,Y) ((X)>(Y)?(X):(Y))
int main()
{
	int a = 10;
	int b = 20;
	int max = MAX(a++, b++);
	printf("max = %d\n", max);
	printf("a = %d\n", a);
	printf("b = %d\n", b);
	return 0;
}
```
那这里打印的结果是什么呢？
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17099895930000wvg8c.png)
为什么会是这样一个结果？
> max被替换为：int max = ( (a++) > (b++) ? (a++) : (b++) )
> a和b都是先使用在++
> > 先a小于b，所以进行后面的(b++)
> > 在进行++，这时后 a=11 b=21
>
到后面的(b++)先使用在++
> > 先使用，赋值给 max，所以 max=21
> > 在++，所以 b=22

# 宏和函数的对比
宏通常被应用与执行简单的运算。
比如在比较两个数中的较大值。

```C
#define MAX(a,b) ((a)>(b)?(a):(b))
```

那为什么不用函数来完成这个认为？原因如下：
> 1. 用于调用函数和从函数返回的代码可能比实际执行这个小型计算工作所需要的时间更多。**所以宏比函数在程序的规模和速度方面更胜一筹。**
> 2. 更为重要的是函数的参数必须声明为特定的类型。所以函数只能在类型合适的表达式上使用。反之这个宏怎可以适用于整形、长整型、浮点型等可以用于>来比较的类型。**宏是类型无关的。**

当然和宏相比函数也有劣势的地方
> 1. 每次使用宏的时候，一份宏定义的代码将插入到程序中。除非宏比较短，否则可能大幅度增加程序的长度。
> 2. 宏是没法调试的。
> 3. 宏由于类型无关，也就不够严谨。
> 4. 宏可能会带来运算符优先级的问题，导致程容易出现错。

宏有时候可以做函数做不到的事情。比如:宏的参数可以出现**类型**，但是函数做不到。
```C
#define MALLOC(num,type) (type*)malloc(jum*sizeof(type))
int main()
{
	int* p = MALLOC(10,int);//使用宏传类型，而不是传值或址
	//替换为
	//int* p = (int*) malloc (10 * sizeof(int));
	return 0;
}
```

# 文件包含
我们已经知道，#include指令可以使另外一个文件被编译就像它实际出现于#inc1ude指令的地方一样。
这种替换的方式很简单∶预处理器先删除这条指令，并用包含文件的内容替换。这样一个源文件被包含10次，那就实际被编译10次。

**头文件包含的方式：**
* 本地文件
```C
#include "test.h"
```

查找策略∶先在源文件所在目录下查找，如果该头文件未找到，编译器就像查找库函数头文件一样在标准位置查找头文件。如果找不到就提示编译错误。linux环境的标准头文件的路径︰
```C
/usr/include
```

**vs环境的标准头文件的路径：按自己的安装路径去找**
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1710162827000vpuynd.png)

* 库文件包含
```C
#include <stdio.h>
```
查找头文件直接去标准路径下去查找，如果找不到就提示编译错误。

## 嵌套包含
**如果胡乱使用文件包含会出现冗余：**
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1710163358000v6cycp.png)

为了防止头文件被多次定义包含，可以编写下面这段代码：
```C
//text.h 头文件中
1. 第一种
#ifndef __TEST_H_ //如果__TEST_H_ 定义过就不执行下面的指令
#define __TEST_H_
int Add(int x , int y );
#endif
//第一次因为没有定义__TEST_H_所以会被定义一次，并执行Add这个函数的指令
//但是后面在包含头文件的时候，__TEST_H_已经在第一次被定义过了，就不会在执行Add这个指令

2. 第二种
#pragma ince
int Add(int x , int y);
//当然这个是现在的一种写法，在vs环境下，当你创建头文件的时候就会自动添加
```
