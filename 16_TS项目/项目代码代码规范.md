# 一、 EditorConfig配置

| editor | 编辑、编辑器 |
| ------ | ------ |

EditorConfig 有助于为不同 IDE 编辑器上处理同一项目的多个开发人员**维护一致的编码风格**。

```bash
// .editorconfig
# http://editorconfig.org  

root = true

[*] # 表示所有文件适用  
charset = utf-8 # 设置文件字符集为 utf-8  
indent_style = space # 缩进风格（tab | space）  
indent_size = 2 # 缩进大小  
end_of_line = lf # 控制换行类型(lf | cr | crlf)  
trim_trailing_whitespace = true # 去除行尾的任意空白字符  
insert_final_newline = true # 始终在文件末尾插入一个新行  
end_of_line = lf #  设置文件换行符为LF（Line Feed，即\n） LF是Unix/Linux和macOS系统默认的换行符

[*.md] # 表示仅 md 文件适用以下规则  
max_line_length = off  # 最大每行包含的字符 
trim_trailing_whitespace = false # 去除尾部空格
```

# 二、配置和插件的关系

简单直接的答案是：**.editorconfig 文件本身只是一个“配置文件”，而 VS Code 插件是一个“解释器”或“执行器”。没有插件，VS Code 就不知道如何读取和遵守这个配置文件里的规则。**

### 1. .editorconfig 文件是什么？

- **它是一个标准**：`.editorconfig` 是一个跨编辑器的代码风格配置文件。它的初衷是让使用不同编辑器（比如 VS Code, WebStorm, Sublime Text 等）的开发者，在同一个项目里能保持基本的代码风格一致。
    
- **它只包含规则，不包含功能**：这个文件里写的都是“要求”，比如：

```ini
[*.{js,jsx,ts,tsx}]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true
```

- 它规定了“JavaScript 和 TypeScript 文件应该用 2 个空格缩进”，但它自身**没有任何能力**去执行这个规定。
    

### 2. EditorConfig for VS Code 插件的作用

这个插件的作用就是**充当 VS Code 和 `.editorconfig` 文件之间的桥梁**。具体来说，它：

- **读取解析**：当你在 VS Code 中打开一个文件时，插件会向上级目录查找 `.editorconfig` 文件，并读取适用于当前文件类型的规则。
    
- **动态配置**：插件会将这些规则**动态地应用**到 VS Code 的编辑器设置上。它会告诉 VS Code：“嘿，当前这个文件，根据项目规定，应该使用 2 个空格缩进，并且要删除行尾空格。”
    
- **覆盖用户/全局设置**：这个插件的配置**优先级很高**。即使你在 VS Code 的用户设置里偏好使用 4 个空格缩进，但只要打开了这个项目，并且有 `.editorconfig` 文件规定用 2 个空格，插件就会强制 VS Code 为这个项目的文件使用 2 个空格。这保证了**项目级别的风格统一**。

### 一个生动的比喻

- **.editorconfig 文件** 就像是一份 **“交通法规”**（例如，限速 60km/h）。
    
- **你的 VS Code 编辑器** 就像是一辆 **“汽车”**。
    
- **EditorConfig for VS Code 插件** 就像是 **“交通法规执法系统”**（交警+摄像头）。
    

如果没有插件（执法系统），你虽然有交通法规（.editorconfig 文件），但司机（VS Code）可能会无视它，按照自己的习惯开车（使用编辑器的默认设置或你的个人设置）。安装了插件后，执法系统就会强制你遵守这里的限速规定（项目代码规范）。

### 为什么需要两者配合？总结

1. **跨编辑器兼容性**：`.editorconfig` 是一个标准，不只是为 VS Code 服务的。你的同事可能用 WebStorm 或 Atom，这些编辑器也有对应的 EditorConfig 插件。大家共用一个配置文件，就能保证风格一致。
    
2. **VS Code 原生不支持**：VS Code 的核心功能并没有内置对 `.editorconfig` 文件的解析能力。它需要通过扩展 API 让插件来实现这个功能。
    
3. **项目约束力**：它确保任何参与项目的人，只要安装了对应的插件，其编辑器行为都会被统一规范。这对于团队协作和代码质量至关重要，避免了因个人编辑器设置不同而产生的“缩进战争”、换行符不一致等问题。
    

**结论：**  
你配置 `.editorconfig` 是**定义规则**，而安装 EditorConfig for VS Code 插件是**让规则在 VS Code 里生效**。两者缺一不可。没有插件，`.editorconfig` 文件在 VS Code 里就只是一堆无用的文本。

# 三、prettier插件

| prettier | 更漂亮的 |
| -------- | ---- |

Prettier 是一款强大的代码格式化工具，支持 JavaScript、TypeScript、CSS、SCSS、Less、JSX、Angular、Vue、GraphQL、JSON、Markdown 等语言，基本上前端能用到的文件格式它都可以搞定，是当下最流行的**代码格式化工具**。

1. 安装prettier

```bash
npm install prettier -D
```

2. 配置.prettierrc文件：

- useTabs：使用tab缩进还是空格缩进，选择false；
    
- tabWidth：tab是空格的情况下，是几个空格，选择2个；
    
- printWidth：当行字符的长度，推荐100，也有人喜欢80或者120；
    
- singleQuote：使用单引号还是双引号，选择true，使用单引号；
    
- trailingComma：在多行输入的尾逗号是否添加，设置为 `none`，比如对象类型的最后一个属性后面是否加一个，；
    
- semi：语句末尾是否要加分号，默认值true，选择false表示不加；


```json title = ".prettierrc"
{  
  "useTabs": false,  
  "tabWidth": 2,  
  "printWidth": 100,  
  "singleQuote": true,  
  "trailingComma": "none",  
  "semi": false  
}
```

3. 创建.prettierignore忽略文件(可以省略)

```json title=".prettierignore"

/dist/*  
.local  
.output.js  
/node_modules/**  
​  
**/*.svg  
**/*.sh  
​  
/public/*
```

### ESLint和prettier联合

ESLint是否启用prettier配置信息进行检查，可以自我进行一个配置：在 `eslint.config.js` 配置文件里面进行配置

#### 不启用prettier

`@vue/eslint-config-prettier/skip-formatting`

这个配置的作用是**完全禁用 Prettier 的格式检查**，只保留 ESLint 的代码质量检查。

```js
// eslint.config.js
import skipFormatting from "@vue/eslint-config-prettier/skip-formatting";

// 在 ESLint 配置中使用
export default {
  extends: [
    // ... 其他配置
    skipFormatting
  ]
}
```

**效果：**

- ESLint 只检查代码质量问题（如未使用的变量、语法错误等）
    
- **不会**根据 `.prettierrc` 配置文件检查代码格式
    
- **不会**因为格式问题（如缩进、分号、引号等）报错

#### 启用prettier

`@vue/eslint-config-prettier`

这个配置会**启用 Prettier 作为 ESLint 规则**，让 ESLint 根据 Prettier 配置检查代码格式。

```js
// eslint.config.js
import prettierConfig from "@vue/eslint-config-prettier";

// 在 ESLint 配置中使用
export default {
  extends: [
    // ... 其他配置
    prettierConfig
  ]
}
```

**效果：**

- ESLint 会**根据 `.prettierrc` 配置文件检查代码格式**
    
- 格式不符合 Prettier 规则时会报 ESLint 错误
    
- 同时也会检查代码质量问题

#### 为什么会有这样的区别？

`skip-formatting` 的设计目的：

1. **分离关注点**：让 Prettier 负责格式化，ESLint 负责代码质量
    
2. **避免重复报错**：防止同一个格式问题被 Prettier 和 ESLint 都报错
    
3. **性能优化**：减少检查时间

#### 总结
简单总结：**第一个跳过格式检查，第二个启用格式检查**。选择哪个取决于你希望 ESLint 在格式检查方面的严格程度。

### 安装所需依赖

根据你的选择安装相应的包：

```bash
# 如果启用 Prettier 检查

# 如果只是禁用格式检查
npm install --save-dev eslint-config-prettier
```


### 启用 Prettier 格式检查

```js
// eslint.config.js
import js from "@eslint/js"
import globals from "globals"
import reactHooks from "eslint-plugin-react-hooks"
import reactRefresh from "eslint-plugin-react-refresh"
import tseslint from "typescript-eslint"
import { defineConfig, globalIgnores } from "eslint/config"
// 新增导入
import prettier from "eslint-config-prettier"
import prettierPlugin from "eslint-plugin-prettier"

export default defineConfig([
  globalIgnores(["dist"]),
  {
    files: ["**/*.{ts,tsx}"],
    extends: [
      js.configs.recommended,
      tseslint.configs.recommended,
      reactHooks.configs.flat.recommended,
      reactRefresh.configs.vite,
      // 添加 Prettier 配置 - 启用格式检查
      prettier,
    ],
    plugins: {
      // 如果还需要其他插件，在这里添加
      "prettier": prettierPlugin,
    },
    rules: {
      // 启用 Prettier 规则
      "prettier/prettier": "error",
      // 可以在这里添加其他自定义规则
    },
    languageOptions: {
      ecmaVersion: 2020,
      globals: globals.browser
    }
  }
])
```

### 不启用 Prettier 格式检查

如果你只想禁用 Prettier 的格式检查，可以使用 `eslint-config-prettier` 的禁用功能：

```js
// eslint.config.js
import js from "@eslint/js"
import globals from "globals"
import reactHooks from "eslint-plugin-react-hooks"
import reactRefresh from "eslint-plugin-react-refresh"
import tseslint from "typescript-eslint"
import { defineConfig, globalIgnores } from "eslint/config"
// 新增导入
import { default as skipFormatting } from "eslint-config-prettier"

export default defineConfig([
  globalIgnores(["dist"]),
  {
    files: ["**/*.{ts,tsx}"],
    extends: [
      js.configs.recommended,
      tseslint.configs.recommended,
      reactHooks.configs.flat.recommended,
      reactRefresh.configs.vite,
      // 明确禁用 Prettier 格式检查
      skipFormatting,
    ],
    languageOptions: {
      ecmaVersion: 2020,
      globals: globals.browser
    }
  }
])
```

> [!warning]
> 
> React 默认就是禁用 Prettier 的格式检查，**ESLint 只检查代码质量问题（如未使用的变量、语法错误等）**，不会根据 Prettier 检查格式错误问题。
