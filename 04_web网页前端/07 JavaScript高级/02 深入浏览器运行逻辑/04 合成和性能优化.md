# 一、特殊解析 - composite合成

绘制的过程，可以将布局后的元素绘制到多个合成图层中。

* 这是浏览器的一种优化手段;

默认情况下，标准流中的内容都是被绘制在同一个图层(Layer)中的;

而一些特殊的属性，会创建一个新的合成层(CompositingLayer)，并且新的图层可以利用GPU来加速绘制;

* 因为**每个合成层都是单独渲染**的;


那么哪些属性可以形成新的合成层呢?常见的一些属性:

* 3D transforms
* video、canvas、iframe
* opacity 动画转换时:
* position: fixed
* wi-change:一个实验性的属性，提前告诉浏览器元素可能发生哪些变化;
* animation 或 transition 设置了opacity、transform;

**合理的使用这个新的合成层可以提高性能：**

例如：移动一个盒子

```html
  <style>
    .box {
      width: 200px;
      height: 200px;
      background-color: orange;
    }

    .container {
      width: 200px;
      height: 200px;
      background-color: red;
      transition: all 1000ms ease-in;
    }
  
    .container:hover {
      transform: translateX(100px);
    }
  </style>

  <div class="box"></div>

  <div class="container"></div>
```

# 二、script元素和页面解析的关系
我们现在已经知道了页面的渲染过程，但是JavaScript在哪里呢?

* 事实上，浏览器在解析HTML的过程中，**遇到了script元素是不能继续构建DOM树的;**
* 它会停止继续构建，**首先下载**JavaScript代码，**并且执行**JavaScript的脚本;
* **只有**等到JavaScript脚本执行结束后，才会继续解析HTML，构建DOM树;

为什么要这样做呢?

* 这是因为JavaScript的作用之一就是操作DOM，并且可以修改DOM;
* 如果我们**等到DOM树构建完成并且渲染再执行JavaScript，会造成严重的回流和重绘，影响页面的性能,**
* 所以会在遇到script元素时，优先下载和执行JavaScript代码，再继续构建DOM树;

但是这个也往往会带来新的问题，特别是现代页面开发中:

* 在目前的开发模式中(**比如Vue**、**React**)，**脚本往往比HTML页面更“重”，处理时间需要更长，**
* 所以会造成页面的解析阻塞，在脚本下载、执行完成之前，用户在界面上什么都看不到;

**为了解决这个问题，script元素给我们提供了两个属性(attribute):[[05 script的defer和async属性]]

# 三、现代浏览器关于script的优化
```html
<body>
  <div class="box">box</div>
  <div class="container">container</div>
  
  <script>
    debugger
  </script>
  <script></script>
  <script></script>
  <script></script>
  
  <div class="bar">我是script后面的bar</div>
</body>
```


![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1730724220000vgyzn6.png)

按照逻辑来说前面在 script 里面模拟了下载script，那么说明我们的 script 是没有下载并且执行完的，那么 **HTML 就是没有解析完没有形成完整的 DOM Tree ，就不会到最后的 Display**

> 但是浏览器为了给用户更加好的体验对这一过程进行了一个优化，不用等 HTML 解析完全,**拿到一部分 DOM Tree 就会先展示出来给用户看到**。
