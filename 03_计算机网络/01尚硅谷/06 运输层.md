# 三、UDP 协议
**用户数据报协议 User Datagram Protocol**

**提供无连接不可靠的传输服务**

* 传输之前不需要建立连接(无连接)，可以想传就传
* 传输过程中出现了误码、丢失等现象时，不要求发送端重传

**适用于对速度要求比较高、对数据质量要求不太高的场景**

* 实时多人游戏、实时音视频聊天等

# 四、TCP协议
**传输控制协议 Transmission Control Protocol**

**提供面向连接的可靠传输服务**

* 传输之前需要先建立连接，传输后需要释放连接
* 出现误码、丢失现象会要求重传，重复的会丢弃，乱序数据也会被正确重组
	* TCP协议会将数据分片传输，不同的数据报通过不同的路径，就会到达目的前后不同，数据就会乱序，TCP协议可以重组数据。

**一般用于要求可靠传输的应用，例如文件、邮件传输**

* TCP 是面向字节流的
* 会给要传输数据的每一个字节都编号

## 4.1 TCP 协议的可靠传输
**ack 的全称是 acknowledgment，意思是确认或确认回应。**

1. 发送方将数据进行编号，发送方和接收方确定允许发送和接收的数据

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715776779000kniu83.png)


2. 发送方将 1~6 和 7~12 个打一个包进行发送，1~6 先到达接收方，但是接收方还没有进行确认

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17157769510008f99kv.png)

> 接收方在对数据进行打包的时候是放到一个发送方的发送缓冲区的。

3. 接收方确定收到后会先发送方发出一个确认号 **`ack（Acknowledgment-确定）= 7`**，意思是下次在发送数据从 7 开始发送，并调节接收的窗口（是一个滑动窗口）

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715777265000ogaepf.png)

> 接收方在接收到数据的时候是放入到接收缓冲区的
> * 如果数据读取了，那么是可以删除的
> * 如果没有读取数据，则不删除

4. 发送方收到接收方收到数据的信息，也移动发送窗口（滑动窗口）

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715777488000jraz7s.png)

5. 刚开始是发送了两个数据包 1~6 和 7~12 ，1~6 已经确定收到，但是 7 ~ 12 还是没有接收到，这个时候如果网络比较好，发送方继续发送 13~18的数据，接收方也确定收到了，但是回复给发送方的确认数据还是 `ack = 7` ，但是会跟着发送一个` 13~18 `收到了，发收方会标记好接收方收到的数据，发送发滑动窗口不进行滑动

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715777797000a9ldt6.png)

6. 如果接收方一段时间还是没有收到 7~12 的数据，超多了发送方的重传计数器的时间，就会触发超时重传。

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17157783570007m5ntm.png)


## 4.2 TCP 协议的首部格式
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715778633000zbz57h.png)

**字段 ACK：**
* 可以延迟发送，收到多个报文的时候，发送一个报文告诉全部收到
* 可以和其他字段合并发送
* 在建立连接后，所发送的数据 ACK 全部都要为 1

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715778912000v4iz8g.png)

**紧急指针：**就是一个有紧急数据的时候进行一个插队

**窗口：**这里的窗口是接收方确认的，是希望发送方根据自己的窗口进行调整。

**MSS：** TCP 协议是要向下链路层进行递交的，链路层最大数据 1500 个字节，不能超过，所以 MSS 规定最大报文断长度。

## 4.3 TCP协议流量控制和拥塞控制
**流量控制：**

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715779352000xezzyx.png)


**拥塞控制：**

* 网络中的链路或路由器过载时，会丢弃数据包，这种情况就是网络拥塞

* 为了避免拥塞，TCP 采用拥塞控制

* 拥塞控制采用一系列方法控制拥塞窗口的大小，进而控制发送窗口的大小

* 发送窗口=min(拥塞窗口，接收窗口)

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17157796970005to2ho.png)

**cwnd ：** 全称是 Congestion Window，意思是拥塞窗口。它是 TCP 协议中的一个参数，用于控制发送方在网络中能发送的数据量，以避免网络拥塞。

**ssthresh ：** TCP 协议中的一个参数，全称是 slow start threshold，意思是慢启动阈值。它是 TCP 连接中用于控制拥塞窗口的重要参数之一。


## 4.4 TCP 连接的建立和释放
**传输 TCP 报文前，需要先建立连接**
* 通过“三报文握手”来建立 TCP 连接

**传输完 TCP 报文，需要释放连接**
* 通过“四报文挥手”来释放 TCP 连接

### 4.4.1 TCP 连接
**TCP 连接是客户端主动发起的**

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715782237000oox0r5.png)


**SYN ：**用于TCP“三报文握手”建立连接，=1时表明这是一个TCP连接报文

**seq：**字节号，发送的第一个数据的字节号
* `seq = x`客户端从`x`字节开始发送数据
* `seq = y`服务器从`y`字节开始发送数据

**ACK：**只有当ACK为1时，确认号字段才有效

**ack：**确认号。意思是收到了可恶端的字节，希望下次发送从 `x+1`个字节发送


### 4.4.2 TCP 释放
**TCP 释放既可以是客户端也可以是服务器端**

**FIN：**用于TCP“四报文挥手”释放连接，`=1`时表明此TCP的发送方已经将全部数据发送完毕，要求释放TCP连接

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715783264000rz53qq.png)


**MSL：** MSL 指的是 `Maximum Segment Lifetime`，即最大分段生命周期。它是 TCP 协议中一个重要的参数，指的是一个 TCP 段`（segment）`在网络中最长的生命周期，即从该段被发送到它被确认或超时的时间。MSL 的典型值是 2 分钟，但这可以根据具体的网络环境和需求进行调整。

如果被动释放连接方法没有收到第 4 段报文，则会重新发一遍第 3 段报文，主动释放连接方再次等待 2MSL。

> [!tip] 特殊情况
> 当被动释放连接方没有数据传输的时候，2、3 报文可以一起进行发送。结果就变成了 3 次挥手。
