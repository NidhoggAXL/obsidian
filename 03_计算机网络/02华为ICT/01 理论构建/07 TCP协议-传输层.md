# 一、TCP定义
TCP 是一种面向连接的、可靠的传输层协议，用于在不可靠的网络（如互联网）上提供可靠的数据传输。

传输层定义了主机应用程序之间端到端的连通性。传输层中最为常见的两个协议分别是传输控制协议TCP(TransmissionControlProtocol)和用户数据包协议UDP(User Datagram Protocol)

# 二、TCP端口号
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17154967190003n4idi.png)


传输控制协议 (TCP) 端口号是 6 位无符号整数，用于标识网络上的特定应用程序或服务。

**众所周知的端口号（0-1023）：**

这些端口号由 Internet Assigned Numbers Authority (IANA) 分配给特定服务。一些常见的众所周知的端口号包括：

- 80：HTTP（Web 浏览）
- 443：HTTPS（安全 Web 浏览）
- 21：FTP（文件传输协议）
- 25：SMTP（简单邮件传输协议）
- 110：POP3（邮局协议 3）

**注册端口号（1024-49151）：**

这些端口号由 IANA 分配给特定应用程序或服务，但没有众所周知的端口号那么常见。

**动态或私有端口号（49152-65535）：**

这些端口号由客户端操作系统动态分配给临时用途。

**一些常见的 TCP 端口号：**

| 服务         | 端口号   |
| ---------- | ----- |
| HTTP       | 80    |
| HTTPS      | 443   |
| FTP        | 21    |
| SSH        | 22    |
| SMTP       | 25    |
| POP3       | 110   |
| IMAP       | 143   |
| DNS        | 53    |
| Telnet     | 23    |
| RDP        | 3389  |
| MySQL      | 3306  |
| PostgreSQL | 5432  |
| Redis      | 6379  |
| MongoDB    | 27017 |

**用途：**

TCP 端口号用于以下目的：

- 标识网络上的特定应用程序或服务
- 将传入连接路由到正确的应用程序或服务
- 限制对某些服务的访问（例如，防火墙规则）

**注意：**

- 某些应用程序或服务可以使用多个端口号。
- 端口号的分配可能会因操作系统或应用程序版本而异。

# 三、TCP报文

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715496859000t3kglr.png)


![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715496371000helnj2.png)

**TCP 首部：**

- **源端口：** 发送方端口号 (16 位)
- **目标端口：** 接收方端口号 (16 位)
- **序号：** 报文段中第一个字节的序列号 (32 位)-做排序
- **确认号：** 发送方期望接收的下一个报文段的序列号 (32 位)-做确认
- **首部长度：** TCP 首部长度，以 32 位字为单位 (4 位)
- **保留：** 保留字段，通常为 0 (6位)
- **标志：** 控制报文段行为的标志位：
    - URG：紧急指针有效
    - ACK：确认号字段有效
    - PSH：推送数据到应用程序
    - RST：重置连接
    - SYN：同步序列号，用于建立连接
    - FIN：结束连接
- **窗口大小：** 接收方可以接收的字节数 (16 位)-控制流量
- **校验和：** 用于检测报文段传输过程中的错误 (16 位)
- **紧急指针：** 如果 URG 标志位为 1，则指定紧急数据的字节偏移量 (16 位)

**数据：**

- 要传输的数据（可变长度）

**TCP 选项（可选）：**

- **最大段大小 (MSS)：** 发送方可以发送的最大报文段大小
- **窗口扩大因子：** 用于扩展窗口大小字段
- **时间戳：** 用于测量网络上的时延和抖动
- **选择确认 (SACK)：** 用于确认报文段的特定部分

**注意：**

- TCP 首部长度可变，具体取决于是否包含 TCP 选项。
- TCP 报文段的最大长度为 65,535 字节（包括 TCP 首部和数据）。

TCP封装在IP内，IP数据报最大长度65535，头部最小20，TCP头部长度最小20，所以最大封装数据长度为65535-20-20=65495

TCP固定首部20个字节，即除去选项及填充
# 四、TCP建立连接的过程
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715497043000blnu91.png)
TCP 建立连接的过程称为 **三次握手**，涉及以下步骤：

**1. 客户端发送 SYN 报文段**

客户端发送一个 TCP SYN 报文段到服务器，其中包含客户端的初始序列号 (ISN)。

**2. 服务器发送 SYN-ACK 报文段**

服务器收到 SYN 报文段后，发送一个 TCP SYN-ACK 报文段作为响应。该报文段包含服务器的 ISN 和对客户端 ISN + 1 的确认号。

**3. 客户端发送 ACK 报文段**

客户端收到 SYN-ACK 报文段后，发送一个 TCP ACK 报文段，确认服务器的 ISN + 1。

**连接建立**

收到客户端的 ACK 报文段后，服务器将连接标记为已建立。客户端也会将连接标记为已建立。

**三次握手的目的：**

- **防止旧连接的混淆：**三次握手可确保只有新的连接才能建立，防止旧连接的混淆。
- **同步序列号：**三次握手允许客户端和服务器同步它们的序列号，以确保可靠的数据传输。
- **避免资源浪费：**三次握手有助于避免在服务器上为未完成的连接分配资源。

**示例：**

以下是三次握手的示例：

```
客户端 -> 服务器：SYN
服务器 -> 客户端：SYN-ACK
客户端 -> 服务器：ACK
```

完成三次握手后，客户端和服务器之间建立了 TCP 连接，可以开始可靠的数据传输。

# 五、TCP传输过程
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715497745000rpgv0q.png)


TCP 传输过程涉及以下步骤：

**1. 建立连接**

在开始数据传输之前，客户端和服务器必须通过 TCP 三次握手建立连接。

**2. 数据传输**

一旦连接建立，客户端和服务器就可以开始交换数据。TCP 将数据分成称为报文段的块，并使用滑动窗口机制来控制数据传输速率。

**3. 确认和重传**

接收方收到报文段后，发送一个 ACK 报文段进行确认。如果发送方在一定时间内没有收到 ACK，它将重传报文段。

**4. 流量控制**

TCP 使用滑动窗口机制来控制数据传输速率。发送方和接收方协商一个窗口大小，该窗口大小指定接收方可以接收的字节数。

**5. 拥塞控制**

TCP 使用拥塞控制算法（如慢启动和拥塞避免）来避免网络拥塞。当网络拥塞时，TCP 会降低其传输速率。

**6. 关闭连接**

数据传输完成后，客户端和服务器通过 TCP 四次挥手关闭连接。

**TCP 四次挥手：**

1. 客户端发送 FIN 报文段，表示它已完成数据发送。
2. 服务器发送 ACK 报文段，确认客户端的 FIN 报文段。
3. 服务器发送 FIN 报文段，表示它已完成数据发送。
4. 客户端发送 ACK 报文段，确认服务器的 FIN 报文段。

完成四次挥手后，TCP 连接关闭。

**示例：**

以下是 TCP 数据传输过程的示例：

1. 客户端和服务器建立 TCP 连接。
2. 客户端发送一个数据报文段到服务器。
3. 服务器收到数据报文段并发送 ACK 报文段进行确认。
4. 服务器发送一个数据报文段到客户端。
5. 客户端收到数据报文段并发送 ACK 报文段进行确认。
6. 客户端和服务器关闭连接。

# 六、TCP流量控制
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/1715498666000hmxqsx.png)

TCP 流量控制是一种机制，用于管理发送方和接收方之间的数据流，以防止网络拥塞。它通过以下方式实现：

**滑动窗口**

- 发送方维护一个滑动窗口，表示允许发送的字节数。
- 接收方维护一个滑动窗口，表示允许接收的字节数。
- 窗口的大小由接收方的缓冲区大小决定。

**确认 (ACK)**

- 接收方发送 ACK 消息，确认已接收的数据。
- ACK 消息包含已接收的最后一个字节的序列号。

**拥塞控制**

- 当网络拥塞时，接收方会发送一个窗口大小为 0 的 ACK 消息。
- 发送方收到窗口大小为 0 的 ACK 消息后，会停止发送数据。
- 接收方缓冲区清空后，会发送一个窗口大小大于 0 的 ACK 消息，允许发送方继续发送数据。

**TCP 流量控制的工作原理**

1. 发送方发送数据，直到滑动窗口已满。
2. 接收方接收数据并发送 ACK 消息。
3. 发送方收到 ACK 消息后，将滑动窗口向前移动已确认的字节数。
4. 如果接收方缓冲区已满，则发送窗口大小为 0 的 ACK 消息。
5. 发送方停止发送数据。
6. 当接收方缓冲区清空后，发送窗口大小大于 0 的 ACK 消息。
7. 发送方继续发送数据。

**优点**

- 防止网络拥塞
- 确保可靠的数据传输
- 优化网络带宽利用率

**缺点**

- 引入延迟，因为发送方必须等待接收方的 ACK 消息
- 可能导致不公平性，因为接收方缓冲区较大的主机可以接收更多数据

# 七、TCP关闭连接
![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2024/17154990130009virrt.png)

TCP 连接关闭是一个受控的过程，涉及发送方和接收方之间的四次握手。它确保所有数据都已传输，并且双方都已准备好终止连接。

**四次握手**

1. **FIN (结束)**：发送方发送一个 FIN 消息，表示它已完成发送数据。
2. **ACK (确认)**：接收方发送一个 ACK 消息，确认已收到 FIN 消息。
3. **FIN (结束)**：接收方发送一个 FIN 消息，表示它已完成发送数据。
4. **ACK (确认)**：发送方发送一个 ACK 消息，确认已收到 FIN 消息。

**过程**

1. 发送方调用 `close()` 函数，向接收方发送一个 FIN 消息。
2. 接收方调用 `recv()` 函数，接收 FIN 消息并发送一个 ACK 消息。
3. 接收方调用 `close()` 函数，向发送方发送一个 FIN 消息。
4. 发送方调用 `recv()` 函数，接收 FIN 消息并发送一个 ACK 消息。

**注意事项**

- 四次握手是半关闭的，这意味着一方可以关闭连接而另一方仍可以继续发送数据。
- 如果一方在收到 FIN 消息后立即关闭连接，可能会导致数据丢失。
- TCP 连接关闭后，双方都必须等待一段时间（称为 TIME_WAIT）才能释放连接资源。

**优点**

- 确保可靠的连接关闭
- 防止数据丢失

**缺点**

- 引入延迟，因为需要四次握手才能关闭连接
