# 一、什么是跨域？

要想理解跨域，要先理解浏览器的同源策略:

 - 同源策略是一个重要的安全策略，它用于限制一个origin的文档或者它加载的脚本如何能与另一个源的资源进行交互。它能帮助阻隔恶意文档，减少可能被攻击的媒介。
 - 如果两个URL的protocol(协议)、port(en-US主机号)(如果有指定的话)和host(端口号)都相同的话，则这两个URL是同源。 
 - 这个方案也被称为“协议/主机/端 - 元组”，或者直接是“元组”

![[跨域问题的产生.excalidraw|700]]


事实上跨域的产生和前端分离的发展有很大的关系:

 - 早期的服务器端渲染的时候，是没有跨域的问题的;
 - 但是随着前后端的分离，目前前端开发的代码和服务器开发的API接 - 往往是分离的，甚至部署在不同的服务器上
 
 这个时候我们就会发现，访问静态资源服务器和API接
 
  - 服务器很有可能不是同一个服务器或者不是同一个端口 。
  - 浏览器发现静态资源和**APl接 - (XHR、Fetch)请求不是来自同一个地方时(同源策略)，就产生了跨域。**

> [!tip] 
> 所以，在静态资源服务器和API服务器(其他资源类同)是同一台服务器时，是没有跨域问题的。

> [!tip] 为什么浏览器会有同源策略？
> 这是为了防止黑客，如果一个网页中被黑客注入了假的页面，这个时候输入的帐号和密码，这些帐号和密码就会放回到黑客的服务器中，这个是非常危险的。所以浏览器就有了同源策略。(黑客的服务器和页面的服务器不是同源的)

# 三、跨域的解决方案

那么跨域问题如何解决呢？

 - 跨域的解决方案几乎都和服务器有关系，单独的前端基本解决不了跨域（虽然网上也能看到各种方案，都是实际开发基本不会使用）。
 - 你说：老师，不对丫，我明明和配置前端的webpack就可以解决跨域问题了。
 - webpack配置的**本质也是在webpack-server的服务器中配置了代理。**

<mark class="hltr-cyan">跨域常见的解决方案：</mark>

 - 方案一：静态资源和API服务器部署在同一个服务器中；
 - 方案二：**CORS**， 即是指跨域资源共享；
 - 方案三：**node代理服务器**（webpack中就是它）；
 - 方案四：**Nginx反向代理**；

不常见的方案：

 - jsonp：现在很少使用了（曾经流行过一段时间）；
 - postMessage：有兴趣了解一下吧；
 - websocket：为了解决跨域，所有的接口都变成socket通信？
 - …..

## 3.2 方案一：CORS

跨源资源共享（CORS， Cross-Origin Resource Sharing跨域资源共享）：

 - 它是一种基于http header的机制；
 - 该机制通过**允许服务器标示除了它自己以外的其它源（域、协议和端口），使得浏览器允许这些 origin 访问加载自己的资源**。

浏览器将 CORS 请求分成两类：简单请求和非简单请求。

### 简单请求

只要同时满足以下两大条件，就属于简单请求（不满足就属于非简单请求）（了解即可）。

请求方法是以下是三种方法之一：

 - HEAD
 - GET
 - POST

HTTP 的头信息不超出以下几种字段：

 - Accept
 - Accept-Language
 - Content-Language
 - Last-Event-ID
 - Content-Type：只限于三个值 application/x-www-form-urlencoded、multipart/form-data、text/plain


### 非简单请求


```js
app.use(async (ctx, next) => {
  // 开启CORS
  ctx.set("Access-Control-Allow-Origin", "*"); //  允许任何来源的域访问，"*"表示所有域
  ctx.set("Access-Control-Allow-Headers", "Accept, Accept-Encoding, Accept-Language,Connection, Content-Type,Host,Origin,Referer,User-Agent,Authorization");// 允许头携带的信息(现代浏览器可以省略)
  ctx.set("Access-Control-Allow-Credentials", true); //  允许跨域请求携带认证信息(如cookies)
  ctx.set("Access-Control-Allow-Methods", "GET,POST,PUT,DELETE,OPTIONS"); //  允许客户端使用的HTTP方法
  if (ctx.method == "OPTIONS") {
    ctx.status = 204
  } else {
    await next();
  }
});
```

这段代码是用于处理跨域请求（CORS，Cross-Origin Resource Sharing）的中间件配置。让我详细解释每个部分：

1. `ctx.set("Access-Control-Allow-Origin", "*")`

- 这行代码设置响应头，允许任何来源的域访问资源
- "`*`" 表示所有域都可以访问，但在生产环境中，建议指定具体的域名以提高安全性

2. `ctx.set("Access-Control-Allow-Headers", ...)`

- 这行代码指定了允许客户端在请求中使用的HTTP头
- 列表包含了常见的请求头，如Accept、Content-Type、Authorization等
- 这些是客户端在发送跨域请求时可能需要使用的请求头

3. `ctx.set("Access-Control-Allow-Credentials", true)`

- 允许跨域请求携带认证信息（如cookies、HTTP认证等）
- 当设置为true时，前端请求也需要设置相应的credentials选项
- 注意：如果设置了credentials为true，则Access-Control-Allow-Origin不能使用"*"

4. `ctx.set("Access-Control-Allow-Methods", "GET,POST,PUT,DELETE,OPTIONS")`

- 指定允许客户端使用的HTTP方法
- 包含了常见的RESTful API方法
- OPTIONS方法特别重要，因为浏览器在发送某些跨域请求前会先发送OPTIONS请求进行预检

5. `if (ctx.method == "OPTIONS")` 处理

- 当请求方法是OPTIONS时（预检请求），直接返回204状态码
- 204表示请求成功但没有返回内容
- 对于其他请求方法，调用await next()继续处理请求

这个中间件的作用是：

1. 处理浏览器的预检请求（OPTIONS）
2. 设置必要的CORS响应头，允许跨域请求
3. 确保跨域请求能够正常携带认证信息
4. 支持常用的HTTP方法

这是一个比较宽松的CORS配置，适合开发环境使用。在生产环境中，建议：

- 将"`*`"改为具体的域名
- 只允许必要的请求头和方法
- 根据实际需求决定是否允许携带认证信息
