# 一、为什么要区分开发和生产环境

首先为什么前面知识点已经写过了development和production 环境了，这里为什么还要编写？先看下面问题

在开发中，一般在开发和生产环境下面使用的是不同的服务器。在开发环境下面会使用测试服务器的地址进行一个测试，在开发环境下面使用真正的服务器。

这是为什么？
在开发的时候前端和后端都会进行各种各样的测试，对服务器进行操作，有时候的一些数据是不希望在生产环境下面出现的，这样会影响用户体验和公司的数据安全。

那么如何解决这个问题呢？

# 二、人为切换

第一种方式：开发人员在**发布到生产环境的时候切换地址**。

```ts
const BASE_URL = "http://codercba.com:1888/dev"//开发环境地址
const BASE_URL = "http://codercba.com:1888/pro"//生产环境地址
```

这种情况下面，依赖认为的在切换，在要发布的时候切换为生产环境地址，**依赖人就是存在太大的不安全。**

# 三、Vite环境变量

Vite 在一个特殊的 import.meta.env 对象上暴露环境变量。这里有一些在所有情况下都可以使用的内建变量：

- `import.meta.env.MODE: {string}` 应用运行的模式。
- `import.meta.env.PROD: {boolean}` 应用是否运行在生产环境。
- `import.meta.env.DEV: {boolean}` 应用是否运行在开发环境 (永远与 import.meta.env.PROD相反)。
- `import.meta.env.SSR: {boolean}` 应用是否运行在 server 上。

```ts
let BASE_URL = ""

//判断是否为开发环境
if (import.meta.env.PROD) {
  BASE_URL = "http://codercba.com:1888/pro"//生产环境地址
} else {
  BASE_URL = "http://codercba.com:1888/dev"//开发环境地址
}
```

> [!abstract] 
> 
> 通过 Vite 提供的内置变量来动态的修改 URL 的地址，这样把切换交给 Vite ，Vite 会根据在开发和生产环境下面动态的修改 URL 地址。

# 四、自定义Vite环境变量

Vite 使用 dotenv 从 **环境目录** 中的下列文件加载额外的环境变量：只有以 `VITE_` 为前缀的变量才会暴露给经过 vite 处理的代码。

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/17617938940005a2stv.png)

例如 `.env.development` 文件的Vite环境变量只有在开发环境下才会加载。

所有定义的变量都会保存到 `import.meta.env` 中，可以获取到 `import.meta.env.VITE_BASE_URL`
