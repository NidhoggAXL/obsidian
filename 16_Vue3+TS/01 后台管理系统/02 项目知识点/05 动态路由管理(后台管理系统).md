# 二、动态管理路由的方式

## 2.1 根据角色

基于角色(Role)的动态路由管理

```ts
const roles = {
 "super": 部分路由,
 "admin": 一部分路由,
 "sever": 少部分路由
}
```

> [!warning] 弊端
> 
> - 每增加一个角色，都要增加 `key: value`
> - 后端返回路由的 json 对象的时候，不好组织 json 对象的结构


## 2.2 根据菜单(menu)

根据在前端的时候，会更具不同的角色返回不同的menu，后面根据不同的菜单对路由进行注册，这样就可以正常时候路由了。

> [!tip] 好处
> 
> - 后端只需要返回一个菜单列表，不需要对注册组件进行抽象为 json
> - 菜单只需要放回路由的路径，路由的组件前端进行路由的注册
> - 有新角色的时候，只是返回一个不同的菜单列表

# 三、创建路由对象和对应组件

#coderwhy/coderhy自动化工具 

当需要使用动态路由管理的时候，需要对每一个路由抽取为一个对象，当菜单有这个这个路由的时候，才对路由进行一个注册。

抽取的对象：（这其实就是一个路由对象）

```ts
const chat = () => import('@/views/main/story/chat/chat.vue')
export default {
  path: '/main/story/chat',
  name: 'chat',
  component: chat,
  children: []
}

```

那么有了路由对象，那么就需要根据创建对应的Vue页面就可以啦。

但是如果存在多个路由对应的页面，在创建的时候就会比较麻烦，需要手动的编写路由对象和对应的页面，那么可以使用 coderwhy 的自动化插件，来自动生成路由对象和对应页面。

> [!abstract]
> 
> 需要保持页面和路由的文件结构基本一致。
> 
> ![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/1762327376000rluu6m.png)

安装自动化工具：

```bash
npm install coderwhy
```

如何使用：

```bash
 coderwhy add3page_setup 文件名 -d src/views/main/文件夹名/文件名
```

例如：

```bash
coderwhy add3page_setup department -d src/views/main/system/department
```

> [!ps]
> 如果router和view的main文件下面不存在system文件夹，会创建文件夹，
> - 在router的system文件夹下面创建一个department文件夹里面包含**department.ts**文件![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/17623278200006bxygf.png)
> - 在view的system文件夹下面创建一个department文件夹里面包含**department.vue**文件![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/176232786000095wv4j.png)

# 四、获取所有router对象

通过上面的操作，把所有的路由对象都放到了一个 .ts 的文件里面，那如何获取到所有的router对象呢？

获取了所有 router 对象后，并保存到一个数组中，这样就可以进行路由的注册：

```ts
const router = createRouter({
  history: createWebHashHistory(),
  routes: [所有的路由对象]
})
```

#vite/import/meta/glob

获取所有的路由对象，就需要对文件进行一个动态加载，在vite中需要使用 `import.meta.glob` 来动态加载文件。

```ts
//动态获取所有文件中的路由对象
const locarRoutes: RouteRecordRaw[] = []
//获取router文件下面的所有ts文件，**是包括子文件
const files = import.meta.glob("../../router/main/**/*.ts")
console.log(files)
```

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/1762330078000726rwb.png)

获取到的是一个对象，以key=value的形式，但是会发现 value 是一个函数，观察知道这是一个[[03 路由懒加载和属性|懒加载(import)]]的函数。如果希望直接获得结构，可以在第二个参数编写 `{ eager: trur}` (渴望的)

```ts
//动态获取所有文件中的路由对象
const locarRoutes: RouteRecordRaw[] = []
//获取router文件下面的所有ts文件，**是包括子文件
const files = import.meta.glob("../../router/main/**/*.ts", { eager: true})
console.log(files)
```

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/1762330416000zjbyfh.png)

这样获取到的 value 就是一个模块，获取到的内容保存到 default 属性中。

在保存的时候会报错：

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/1762330655000h17a2t.png)

这是因为 files 的类型有问题：推断中存在 unknown 类型，所有要消除这个 unknown 累心，为什么会是这个呢？模块的话ts无法自行推导，所以显示编写为 any类型。

![gh](https://raw.githubusercontent.com/AXLflechazoPN/Obsidian/main/2025/17623306820004u8sd8.png)

```ts
//5
//动态获取所有文件中的路由对象
//locarRoutes的类型是根据createRouter中的routes编写的
const locarRoutes: RouteRecordRaw[] = []
//获取router文件下面的所有ts文件，**是包括子文件
const files:Record<string, any> = import.meta.glob("../../router/main/**/*.ts", { eager: true})
for (const key in files) {
const module = files[key]
locarRoutes.push(module.default)
}
console.log(locarRoutes)
```
